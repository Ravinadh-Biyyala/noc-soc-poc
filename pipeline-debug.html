① DATA ENGINEER — bronze_to_silver
executor: makeDataEngineerExecutor(projectId, log) · model: gpt-4.1-mini · maxIter: 6
get_schema_info(tableName)
→ information_schema.columns for proj_{id}_raw.{tableName}
Returns: [{column_name, data_type, is_nullable}]
profile_data(tableName, columns?)
→ COUNT(*), NULL count, DISTINCT count, MIN, MAX per column
Returns: {tableName, rowCount, columns: [{name, nullCount, distinctCount, min, max}]}
propose_cleaning(title, sql, kind, targetTableName)
→ INSERT INTO project_transformations (status="proposed")
kind ∈ {cleanse, join, aggregate, view, rename}
Validates: SQL syntax, kind matches DDL type
execute_transformation(transformationId)
→ Resolves upstream dependencies recursively (depth ≤5)
→ BEGIN; DROP IF EXISTS; CREATE TABLE/VIEW; COMMIT
→ Strips CREATE OR REPLACE (Postgres bug workaround)
System prompt includes:
• Raw table list with col types & row counts
• SQL safety rules (warehouse-only writes)
• Transform DDL rules by kind
✗ No CHART_RULES (Phase 1 never emits visuals)
② DATA MODELER — silver_to_semantic
Two executors: makeSemanticModelExecutor + makeDashboardExecutor · model: gpt-4.1-mini · maxIter: 6
list_warehouse_tables()
→ information_schema.columns for proj_{id}_warehouse
Returns: [{tableName, columns: [{name, type}]}]
propose_star_schema(facts[], dims[], confidenceRatio)
→ In-memory state holder (not DB yet)
Agent calls this once, then generate_semantic_graph
generate_semantic_graph(facts, dims, joins)
→ INSERT project_semantic_models (status="proposed")
Falls back to in-memory starSchema if args omitted
execute_warehouse_query(sql)
→ SELECT-only (assertSelectOnly validates)
→ Must stay in proj_{id}_warehouse schema
Returns data for chart building
create_dashboard(title, description, charts[])
→ INSERT user_dashboards + dashboard_charts
→ Auto-generates 4 KPI stat cards via ensureProjectKpis()
→ Normalizes charts: drops KPIs, infers xKey/yKey
Two prompt variants:
• Semantic pass: classify tables → propose_star_schema → generate_semantic_graph
• Dashboard pass: design 5-6 visuals → execute_warehouse_query → create_dashboard
✓ CHART_RULES included (emits [CHART:{...}] blocks)
③ METRIC ARCHITECT — gold_metrics
executor: makeMetricArchitectExecutor(projectId) · model: gpt-4.1-mini · maxIter: 6
read_semantic_model()
→ SELECT project_semantic_models WHERE status="applied"
Returns: {facts[], dims[], joins[]}
list_warehouse_columns()
→ Warehouse schema grouped by table
Used to identify candidate metric columns
suggest_metrics(columnHints)
→ Pattern-match: revenue*, cost*, qty*, claim*, loss*, premium*
→ Cross-column suggestions: rev−cost=profit, (rev−cost)/rev=margin%
Lightweight — LLM uses as inspiration
save_measure_metadata(metricName, sqlFormula, dependsOnTables, description)
→ INSERT/UPDATE project_metrics (dedupes by metricName)
→ assertMeasureFormula: rejects DDL/DML, semicolons
Returns {id, status: "proposed"}
System prompt includes:
• Numeric column hints (for metric building)
• Existing metrics list (deduplication)
• Hard rules: no DDL, no semicolons, no materialised measures
✗ No CHART_RULES, no transformation DDL

Structured Pipeline (pipeline/graph.py) — with native LangGraph interrupts

Key difference from TS: Uses LangGraph native interrupt() — Python async context supports blocking I/O. State persisted by AsyncPostgresSaver. Thread key: pipeline:{project_id}

Auto-Pipeline (auto_pipeline/graph.py) — 5-Lens Parallel Analysis

LangGraph pattern: 5 parallel nodes all point to visualization. LangGraph uses the 5 incoming edges as a barrier — waits for all to complete before firing. Annotated[dict, merge_dict] reducer allows concurrent writes without conflicts.
